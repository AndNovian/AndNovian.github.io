Have you ever set up Defender policies from GPO or Intune, and wondered whether the policies is indeed applied or not? Or maybe you are using MDE, and is seeing Security Recommendation to Enable Defender ASR Rules, you have applied it through GPO/Intune, but the Recommendation won’t go away?

This guide will be focused at explaining Defender logs called MpRegistry.txt, that will show you how Defender is receiving the policy, and applied policy. This guiide will help you troubleshoot the following scenarios:

  1. You have applied Defender Policy (such as Real Time Protection, Cloud Protection, or ASR Rules), but unsure if the device received the policy or not.
  2. You have applied Defender Policy , but it is not being enabled/disabled per your config.
  3. You try to enable/disable Defender policy, but it says config is locked by administrator.
  4. You have applied Defender Policy, but MDE Security Recommendation is still there and won’t go away.

Has the Defender policy reached the device?

First step of a policy to be applied is that, it need to be arrived on the Device. Before Defender receive and apply it.

Management Tools > Device/Endpoint > Defender

Next, we can reverse the flow to check the logs to see if configuration is applied/arrived at each points.

Defender (MpRegistry.txt) > Device/Endpoint (GPResult/MDM Logs/etc) > Management Tools (Respective tools logs)

Collecting Defender MpRegistry logs

You can collect Defender logs, locally using the following command:

"%programfiles%\Windows Defender\mpcmdrun.exe" -getfiles

Once collection finish, you will see the output path. Extract the cab file to find the MpRegistry.txt
